快速开始

10 分钟接入

按以下三步即可完成大转盘游戏的完整接入，从引入脚本到响应中奖结果全部覆盖。

1
引入组件
html 
<!-- 方式一：CDN 引入（推荐）-->
<script src="https://cdn.gamehub.io/v1/gamehub.min.js"></script>

<!-- 方式二：npm 安装 -->
npm install @gamehub/mini-games
2
初始化游戏
javascript 
const game = GameHub.create('lucky-wheel', {
  container: '#game-container',
  prizes: [
    { id: 'prize_1', name: '100元优惠券', probability: 0.05 },
    { id: 'prize_2', name: '谢谢参与',    probability: 0.50 },
    // ... 更多奖品
  ],
  onSpin: async () => {
    // 调用你的后端接口获取结果
    const result = await fetch('/api/lottery/spin');
    return result.json(); // { prizeId: 'prize_1' }
  },
  onComplete: (prize) => {
    console.log('中奖了：', prize.name);
  }
});
3
触发抽奖
javascript 
// 用户点击后手动调用
game.spin();

// 或者让组件自动处理点击
game.mount(); // 自动绑定点击事件
✅ 三步完成后刷新页面，大转盘即可正常渲染并响应抽奖操作。如需深度定制样式，参考 theme 和 customStyles 配置项。

游戏组件 API

大转盘 LuckyWheel

经典概率转盘，支持自定义格数、主题色、动效速度与保底策略。

配置参数
参数名类型必填默认值说明
container string | Element 必填 — CSS 选择器或 DOM 元素
prizes Prize[] 必填 — 奖品列表，见下方 Prize 结构
theme string 否 'default' 主题：default / red / ocean / gold
dailyLimit number 否 3 每日抽奖次数上限，0 表示不限
guarantee object 否 null 保底配置，详见奖池配置章节
onSpin async function 必填 — 抽奖回调，需返回 { prizeId }
onComplete function 否 — 结果动画播放完成回调，参数为 Prize 对象
onError function 否 — 错误处理，参数为 Error 对象
Prize 对象结构
typescript 
interface Prize {
  id:          string;   // 奖品唯一 ID
  name:        string;   // 奖品名称
  probability: number;   // 概率 0–1，所有奖品之和应 ≤ 1
  stock?:      number;   // 库存，不传则不限
  emoji?:      string;   // 显示图标，如 '🎁'
  color?:      string;   // 格子背景色，如 '#7C3AED'
}
Methods 实例方法
方法参数返回值说明
spin() — Promise<Prize> 触发一次抽奖，异步返回中奖奖品
reset() — void 重置状态（次数/动画），不销毁实例
updatePrizes(prizes) Prize[] void 动态更新奖品列表，实时生效
destroy() — void 销毁实例，解绑所有事件，释放内存
Events 事件
事件名触发时机回调参数
spin-start 转盘开始旋转时 —
spin-end 转盘停止、后端结果返回后 { prize: Prize }
prize-claimed 用户确认领奖弹窗后 { prize: Prize, serialNo: string }
error 抽奖请求失败或超时 { code: number, message: string }

奖池配置

奖品结构

每个奖品是一个 Prize 对象，最少提供 id、name、probability 三个字段。建议同时配置 stock 以防止热门奖品超发。

javascript 
const prizes = [
  { id: 'p1', name: 'iPhone 15',  probability: 0.001, stock: 5,   emoji: '📱' },
  { id: 'p2', name: '100元券',  probability: 0.05,  stock: 200, emoji: '🎟'  },
  { id: 'p3', name: '10元券',   probability: 0.15,  stock: 500, emoji: '🎫'  },
  { id: 'p4', name: '谢谢参与', probability: 0.799              },
];
💡 所有奖品概率之和建议等于 1.0。若总和小于 1，剩余概率会自动分配给概率最高的奖品（通常是"谢谢参与"）。

概率规则

GameHub 采用服务端加权随机算法确定最终奖品，前端只负责播放动效，杜绝客户端作弊。概率计算流程如下：

  1. 服务端按 probability 权重生成随机数，命中对应奖品区间
  2. 检查该奖品库存，若已售罄则按概率降级到下一个有库存奖品
  3. 触发保底检查（如已配置），条件满足时强制覆盖为保底奖品
  4. 返回最终 prizeId 给前端，前端播放对应格子的动效
⚠️ 前端传入的 probability 仅用于本地 Demo 模式预览；生产环境中实际中奖结果完全由后端决定，前端概率配置不影响真实发奖。

保底机制

为提升用户留存，可配置连续 N 次未中指定奖品后强制触发保底：

javascript 
guarantee: {
  enabled:      true,
  triggerAfter: 5,          // 连续 N 次未中大奖后触发
  prizes:      ['prize_1', 'prize_3']  // 保底奖品范围（随机命中其一）
}

库存控制说明

配置 stock 字段后，每次发奖自动扣减库存。库存归零时该奖品自动下架，抽奖结果降级为备用奖品（概率最高且有库存的奖品）。建议对高价值奖品始终配置 stock，避免超发风险。

后端对接

抽奖接口规范

GameHub 前端通过 onSpin 回调调用业务方自有后端，接口格式建议遵循以下规范以便使用中台提供的签名验签工具。

▸ 请求（前端 → 后端）

json 
POST /api/lottery/spin
{
  "gameId":    "activity_001",
  "userId":    "user_123",
  "gameType":  "lucky-wheel",
  "timestamp": 1720000000,
  "sign":      "md5(gameId + userId + timestamp + secret)"
}

▸ 响应（后端 → 前端）

json 
{
  "code": 0,
  "data": {
    "prizeId":      "prize_1",
    "prizeName":    "100元优惠券",
    "serialNo":     "LT20260706001234",
    "remainChance": 2
  }
}
错误码说明
code含义前端处理建议
0 成功 播放中奖动效
1001 今日次数已用完 展示「明日再来」提示，禁用按钮
1002 库存不足 降级展示备用奖品，不报错
1003 签名错误 检查 secret 配置，不暴露给用户
1004 活动未开始 展示活动倒计时
1005 活动已结束 展示活动结束页

回调协议

中奖后如需异步通知业务后端（如发放优惠券、积分），可配置 callbackUrl，GameHub 服务端将在用户确认领奖后主动推送：

json 
POST {callbackUrl}
{
  "event":     "prize.claimed",
  "gameId":    "activity_001",
  "userId":    "user_123",
  "prizeId":   "prize_1",
  "serialNo":  "LT20260706001234",
  "timestamp": 1720000100,
  "sign":      "md5(gameId+userId+serialNo+timestamp+secret)"
}
✅ 业务后端收到回调后，需在 5 秒内返回 HTTP 200 及 {"code":0}，否则 GameHub 将按指数退避重试，最多 5 次。

防作弊说明

1
结果由后端决定 — 前端 onSpin 回调必须请求业务后端，由服务端完成随机抽奖并返回 prizeId。禁止在前端 JS 中直接决定中奖结果，否则用户可通过修改 JS 变量获取任意奖品。
2
时间戳签名验签 — 每次抽奖请求携带 timestamp 和 sign（HMAC-MD5），后端验证签名有效且时间戳在 ±60 秒内，防止重放攻击。secret 务必保存在服务端，不得写入前端代码。
3
接口限流 — 建议对 /api/lottery/spin 接口按用户 ID 限制 QPS（推荐 1 次/秒），同时在数据库层对 serialNo 加唯一索引，确保同一请求不被重复发奖。

常见问题

Q
转盘指针停止后没有弹出中奖弹窗，是什么原因？
通常是 onSpin 回调返回的 prizeId 与 prizes 列表中的 id 不匹配。请检查后端返回值是否与前端配置的奖品 ID 完全一致（区分大小写）。也可临时开启 debug: true 选项，控制台会输出详细匹配日志。
Q
每日次数限制是前端控制还是后端控制？能否绕过？
dailyLimit 参数仅做前端 UI 层拦截（禁用按钮、展示提示），不可作为安全保障。真正可信的次数校验必须在后端实现：后端收到请求时先查询该用户当日已抽次数，超限则返回错误码 1001。前端限制可清除 localStorage 绕过，请务必不要省略后端校验。
Q
能在微信小程序中使用吗？有哪些限制？
支持微信小程序，需通过 npm 方式接入（npm install @gamehub/mini-games），在小程序开发者工具中执行「构建 npm」后即可使用。注意：Canvas 2D 模式在真机上性能更好，刮刮乐组件会自动检测环境并使用 WebGL 降级方案。陀螺仪摇骰子需在 app.json 中申请 scope.deviceMotion 权限。
Q
如何自定义转盘/格子的样式，不想用内置主题？
在 Prize 对象上设置 color（格子背景色）和 emoji（图标）可完成基础定制。需要深度定制（自定义字体、边框、指针图片等），使用 customStyles 配置对象，支持传入 CSS 变量覆盖。极端情况可设置 renderMode: 'custom'，由组件仅负责概率逻辑，渲染层完全由业务方接管。
Q
回调通知失败了，用户已看到中奖弹窗，奖品还会发吗？
GameHub 服务端会按指数退避重试回调，最多 5 次（间隔约 1s→2s→4s→8s→16s）。若全部失败，该 serialNo 会进入待处理队列，可在管理后台手动触发补发或查看详情。建议业务后端实现幂等处理：以 serialNo 为唯一键，重复回调时直接返回成功而不重复发奖。
