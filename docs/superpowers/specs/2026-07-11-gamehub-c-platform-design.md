## 一、产品定位与愿景

### 1.1 定位

**C端消费者游戏创造平台**，对标微信小游戏（轻量社交裂变）+ 休闲游戏合集（统一账号/成就/货币体系）+ CrazyGames/Poki（网页即用，免安装）。

### 1.2 核心卖点

**零代码游戏创造。** 用户既是玩家也是创造者，可以自己搭建游戏乐园。不需要懂代码，通过「画布编辑器 + AI 辅助」就能配置出独一无二的游戏，然后分享给朋友玩。

### 1.3 与现版本的差异

| 维度 | 当前版本 (1.0) | 目标版本 (2.0) |
|------|---------------|---------------|
| 游戏逻辑 | 硬编码在 51 个独立 HTML 文件中 | Block 配置驱动，通用引擎动态渲染 |
| 用户角色 | 纯玩家 | 玩家 + 创造者 |
| 配置能力 | builder.html 只能配奖池/概率 | 画布编辑器，游戏规则/内容/主题全可调 |
| 后端 | campaigns（B端活动概念） | user_configs（C端创作概念） |
| 增长模式 | 无 | 分享裂变 + Fork + 模板市场 |

## 二、路线决策

### 先孤岛后乐园

| 阶段 | 说明 |
|------|------|
| **孤岛模式（现在）** | 用户可创建多个独立游戏，每个有独立的分享链接，互不相干。类似 Google Docs，一个游戏一个链接 |
| **乐园模式（未来）** | 用户把多个游戏串联成一个主题乐园，访客按顺序体验，跨游戏数据流转（金币/排名共享） |

### 游戏配置深度分级

| 层级 | 数量 | 说明 | 配置方式 |
|------|------|------|----------|
| 旗舰深度配置 | 8 个 | 大富翁、消消乐、2048、答题、大转盘、贪吃蛇、飞行棋、真心话大冒险 | Block 模式（类 Notion） |
| 标配 | ~25 个 | 开放 10-20 个核心参数 | 简单参数面板 |
| 轻配 | ~18 个 | 开放 3-5 个参数（奖品/概率/难度） | 极简表单 |

**策略：** 先做透 8 个旗舰的 Block 深度配置，跑通完整链路后扩展到其余 43 个。

### 8 个旗舰游戏选择理由

- **大富翁**：回合制 + 棋盘 + 建筑 + 卡牌，Block 类型最全，天然是样板
- **消消乐**：网格 + 消除逻辑，考验 Block 引擎的规则表达能力
- **2048**：网格 + 合并逻辑，经典益智
- **答题**：题库 + 计分，考验 Collection Block 的题库管理
- **大转盘**：概率抽选，最基础的抽奖模式
- **贪吃蛇**：实时操作 + 碰撞，考验引擎的实时渲染能力
- **飞行棋**：棋盘 + 多玩家 + 随机骰子，社交类代表
- **真心话大冒险**：卡牌堆 + 规则，新增游戏，卡牌内容创作场景

## 四、Block 系统设计

### 4.1 Block 是什么

每个 Block 是一个**可配置的游戏元素**，有类型、有属性、有默认值。用户可以增删改排序。

### 4.2 9 种通用 Block 类型

| # | 类型 | 说明 | 核心配置项 | 适用场景 |
|---|------|------|------------|----------|
| 🗺️ | **Grid** | 网格/棋盘 | 尺寸、每格类型和属性 | 大富翁棋盘、消消乐棋盘、飞行棋棋盘、2048网格 |
| 🎲 | **Collection** | 卡堆/奖池/题库 | 条目、权重、抽选模式（随机均匀/按权重/顺序循环） | 机会卡、大转盘奖池、问答题库、真心话牌堆 |
| 📐 | **Rule** | 数值规则 | 初始值、上下限、计算公式 | 金币初始值、步数限制、时间限制、破产线 |
| 🧩 | **Piece** | 实体/棋子 | 造型、移动规则、碰撞检测 | 2048方块、贪吃蛇节点、飞行棋棋子 |
| 🏗️ | **Building** | 可建造/升级结构 | 费用、升级链、效果 | 大富翁建筑、飞行棋基地 |
| 🏪 | **Store** | 商店/交易 | 货币类型、商品列表、价格 | 大富翁建筑商店、飞行棋道具商店 |
| ⚡ | **Effect** | 触发效果 | 触发条件、动作、数值 | 踩格子触发、跳过惩罚、连消奖励 |
| 👥 | **Player** | 玩家设置 | 人数、阵营、初始资源 | 多人模式、阵营划分 |
| 🎨 | **Theme** | 视觉主题 | 配色方案、字体、风格预设 | 所有游戏的视觉定制 |

### 4.4 Block 数据结构

用户保存的配置就是一个 Block 数组，存储为 JSONB：

```json
{
  "id": "conf_abc123",
  "gameType": "monopoly",
  "title": "我的赛博大富翁",
  "createdBy": "user_xxx",
  "isPublic": true,
  "blocks": [
    {
      "id": "blk_001",
      "type": "grid",
      "label": "棋盘",
      "config": {
        "size": 24,
        "cells": [
          { "index": 0, "type": "start", "label": "起点", "effect": { "gold": 200 } },
          { "index": 1, "type": "property", "label": "中关村", "price": 800, "rent": [50, 150, 400] }
        ]
      }
    },
    {
      "id": "blk_002",
      "type": "collection",
      "label": "机会卡",
      "config": {
        "drawMode": "random",
        "cards": [
          { "id": "c1", "title": "前进5格", "effect": { "move": 5 } }
        ]
      }
    }
  ]
}
```

### 5.2 关键交互

- **左侧 Block 列表**：可拖拽排序、开关显示/隐藏、删除。底部「+ 添加 Block」弹出 Block 类型选择器
- **中间实时预览**：iframe 沙箱隔离，改配置即时刷新，所见即所得
- **右侧属性面板**：点击左侧 Block 或预览中的元素自动定位，根据 Block 类型自动渲染编辑组件
- **AI 面板固定在底部**：自然语言输入 → AI 分析意图 → 修改对应 Block → 预览即时刷新
- **模板市场入口**：顶部导航，基于别人的公开配置 fork 开始改

### 5.3 AI 助手能力边界

- ✅ **帮你改配置**：「把棋盘改成32格」「增加一种叫酒店的超级建筑」，直接操作 Block 属性
- ✅ **帮你原创内容**：「帮我设计一个丧尸主题大富翁」，自动填充所有格子的名字、建筑名字、卡牌文案，全部符合丧尸世界观
- ❌ **不写新游戏代码**：AI 不能创造新的游戏类型（如五子棋），只能修改已有的 8 款游戏的 Block 配置

## 六、游戏运行时引擎

### 6.1 引擎架构

**核心原则：引擎不变，配置可变。** 不是写 8 个引擎，是写 1 个通用引擎 + 8 个游戏核心逻辑文件。

```
runtime/
├── engine.js          ← 通用引擎：读配置、调度Block、事件总线
├── core/
│   ├── grid.js        ← 棋盘渲染器（大富翁/飞行棋/消消乐/2048 共用）
│   ├── collection.js  ← 卡堆/奖池抽选器（大转盘/真心话/答题 共用）
│   ├── piece.js       ← 实体控制器（2048/贪吃蛇 共用）
│   ├── store.js       ← 商店交易器（大富翁/飞行棋 共用）
│   ├── effect.js      ← 效果触发器（所有游戏共用）
│   ├── player.js      ← 玩家管理器
│   └── theme.js       ← 主题渲染器
└── games/
    ├── monopoly.js    ← 大富翁专用逻辑（回合制、破产判定）
    ├── snake.js       ← 贪吃蛇专用逻辑（移动、碰撞、增长）
    └── ...8 个文件
```

### 6.2 渲染流程（大富翁为例）

```
1. engine.js 加载用户配置 JSON
2. 遍历 blocks 数组：
   ├── grid block      → grid.js 渲染棋盘（Canvas 绘制格子）
   ├── store block     → store.js 创建建筑商店
   ├── collection block → collection.js 创建机会卡堆
   ├── effect block    → effect.js 注册事件监听
   ├── rule block      → 设置初始金币/玩家数
   └── theme block     → theme.js 应用赛博朋克配色
3. monopoly.js 接管回合循环（掷骰子→移动→结算→下一人）
4. 游戏结束 → engine.js 上报成绩
```

### 6.3 Block 核心模块设计原则

- **Block 核心模块是纯函数**——`grid.js` 不关心是哪个游戏在用，只管「给我配置，我给你棋盘」
- **游戏专用逻辑放在 games/ 下**，只处理该游戏独特的流程
- **新游戏接入成本低**：只需写 1 个 game 文件 + 复用已有的 Block 核心模块

## 七、数据存储设计

### 7.1 新增表

**user_configs — 用户游戏配置**

```sql
create table user_configs (
  id            uuid primary key default gen_random_uuid(),
  user_id       uuid references players(id) on delete cascade,
  game_type     text not null,           -- monopoly/snake/2048/truth-or-dare...
  title         text not null,           -- "我的赛博大富翁"
  blocks        jsonb not null,          -- Block 配置数组
  is_public     boolean default false,   -- 公开（允许 fork）
  is_published  boolean default false,   -- 已发布（生成分享链接）
  share_slug    text unique,             -- 分享短码
  fork_from     uuid references user_configs(id),
  plays_count   integer default 0,
  version       integer default 1,
  created_at    timestamptz default now(),
  updated_at    timestamptz default now()
);
```

**block_templates — Block 模板市场**

```sql
create table block_templates (
  id            uuid primary key default gen_random_uuid(),
  game_type     text not null,
  name          text not null,           -- "经典20格棋盘" "情侣真心话牌堆"
  block_type    text not null,           -- grid/collection/store/effect...
  description   text,
  config        jsonb not null,          -- 单个 Block 的预设配置
  author_id     uuid references players(id),
  usage_count   integer default 0,
  created_at    timestamptz default now()
);
```

### 7.2 旧表改造

| 旧表 | 改造 |
|------|------|
| `clients` | **废弃。** C 端不需要 B 端客户概念 |
| `campaigns` | **废弃。** 活动概念被 `user_configs` + 分享链接取代 |
| `players` | **保留。** 加 `display_name`、`avatar_url`、`bio` 字段 |
| `play_sessions` | **保留。** 加 `config_id uuid references user_configs(id)` 字段 |
| `campaign_stats` | **改为 game_stats** — 按 game_type + config_id 聚合统计 |

### 7.3 存储量估算

一个深度配置的大富翁：~15 个 Block × 每个约 2KB = **~30KB/配置**。Supabase 免费额度 500MB ≈ **~17,000 个配置**，前期完全够用。

## 八、分享与裂变系统

### 8.1 分享链路

```
创作者发布游戏
    │
    ├── ① 分享链接: gamehub.io/play/x1a2b3
    │       └── 自动生成 OG 元标签（标题+描述+封面图）
    │
    ├── ② 嵌入代码: <iframe src="gamehub.io/play/x1a2b3"></iframe>
    │       └── 可嵌入博客、Notion、飞书等
    │
    ├── ③ 社媒战绩卡: Canvas 绘制 + 二维码
    │       └── "我在这局大富翁赢了5000金币！"
    │
    └── ④ 模板市场: 公开配置出现在「发现」页
            └── 一键 Fork，改改发布
```

### 8.2 裂变机制

| 机制 | 做法 |
|------|------|
| 🔗 **邀请追踪** | 分享链接带 `?invite=user_xxx`，双方各得金币 |
| 🏆 **排行榜传播** | 好友互晒排名，点击卡片直接进入游戏 |
| 🎨 **Fork 树** | 类似 GitHub fork 关系图，展示创作影响力 |
| 📊 **创作统计** | 配置被玩次数、被 Fork 次数公开显示 |

### 8.3 技术要点

- `share_slug`：8 位随机字符，去重后写入
- OG Image：Canvas 绘制逻辑在 Edge Function 服务端渲染
- 嵌入安全：iframe sandbox 属性限制

## 十、AI 助手设计

### 10.1 能力边界

- ✅ **改配置**：「把棋盘改成32格」「增加酒店建筑」
- ✅ **原创内容**：「设计丧尸主题大富翁」，自动填充所有文案、名字、卡牌内容
- ❌ **不写新游戏代码**

### 10.2 工作原理

```
用户输入自然语言
      │
      ▼
AI 分析意图 → 匹配目标 Block → 生成配置变更
      │
      ▼
应用变更到编辑器 → 实时预览刷新 → 用户确认或撤销
```

### 10.3 交互位置

编辑器底部固定栏，类似 Cursor Composer。用户可以随时调出，也可以忽略纯手动操作。

## 十二、关键设计决策记录

| # | 决策 | 原因 |
|---|------|------|
| 1 | 先做 8 个旗舰而非 51 个全深度 | 51 个全深度工程量过大，8 个旗舰跑通后模式可复制 |
| 2 | Block 模式而非简单参数表 | 参数表无法表达「增删格子和卡牌」，Block 模式才能让用户真正创造 |
| 3 | 9 种通用 Block 覆盖全部 | Deck 和 Pool 合并为 Collection，补了 Store 和 Effect 两个缺失类型 |
| 4 | 1 个通用引擎 + 8 个 game 文件 | 最大化复用 core 模块，新游戏接入成本低 |
| 5 | 三栏画布编辑器 | 左侧列表 + 中间预览 + 右侧属性，Figma/Webflow 验证过的成熟模式 |
| 6 | AI 只改配置不写代码 | 保证质量可控，降低 AI 出错风险 |
| 7 | 先孤岛再乐园 | 孤岛是基础和刚需，乐园是锦上添花，分步降低风险 |
| 8 | 旧 51 个 HTML 逐步废弃 | 保留为参考和过渡期使用，新架构稳定后下线 |
